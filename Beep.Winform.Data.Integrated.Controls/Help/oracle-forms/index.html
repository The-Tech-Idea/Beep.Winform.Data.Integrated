Oracle Forms Emulation in BeepForms

This document describes the Oracle Forms-compatible surface added to TheTechIdea.Beep.Winform.Data.Integrated so that BeepForms, BeepBlock, and BeepDataConnection can host forms authored against the Oracle Forms mental model — built-ins (GO_BLOCK, COMMIT, ENTER_QUERY, …), query-by-example mode, a modal logon dialog, and the standard When-New-Form-Instance trigger.

Status: All planned phases (0, 1, 2, 5C) are implemented and build green: 0 errors in BeepDM and Beep.Winform.Data.Integrated.Controls.

Phase 0 — Design-time BeepDataConnection

0a — BeepDataConnectionDesigner

New ComponentDesigner in TheTechIdea.Beep.Winform.Controls.Design.Server that satisfies the [Designer] forward reference on BeepDataConnection and, when the control is dropped onto a form, acquires a design-time IBeepService lease via DesignTimeBeepServiceManager.Acquire(component) and calls AttachSharedBeepService(...) so the property grid shows live connections from the host BeepService.

0b — Bare-catch logging

Three catch {} blocks in BeepDataConnection (initialise BeepService, hydrate driver catalog, save) were replaced with logged catches using System.Diagnostics.Trace so designers can see why a connection list failed to load.

Phase 1 — Built-ins

1a — Portable contracts (BeepDM)

  • IBeepBuiltins — the 18-entry built-in surface.
  • IBuiltinHost — what a built-in needs from its host (active block/item, mode mutation, property access, trigger fan-out).
  • BeepBuiltinException — 6-arg ctor with errorCode, builtinName, blockName, itemName, message, inner; default FRM-40735 on rethrows.

1b — Concrete impl BeepBuiltins

Implements IBeepBuiltins against an IBuiltinHost. Each built-in wraps the call in RunWithTrigger / RunWithTriggerAsync which raises TriggerExecuting / TriggerExecuted so form-level and block-level handlers can intercept.

Phase 2 — Enter-Query mode

2a — DataBlockMode extended

The enum was extended from 2 values to 6 — additive, source-compatible with serialised blocks: 

public enum DataBlockMode
{
    Normal     = 0,   // default CRUD / data-display
    EnterQuery = 1,   // user is typing criteria (example record)
    Query      = 2,   // transitional (kept for legacy)
    CRUD       = 3,   // explicit alias of Normal (preserved)
    ReadOnly   = 4,
    Insert     = 5
}

2b — Example-record presenter

BeepBlockViewState.IsQueryMode is now Mode == EnterQuery || Mode == Query. The lifecycle:

  1. ENTER_QUERY → Mode = EnterQuery, the block enters the example-record criterion state.
  2. EXECUTE_QUERY → runs the query, on success transitions EnterQuery → CRUD.
  3. EXIT_QUERY → EnterQuery → CRUD without running the query.

Phase 5C — Logon dialog

WinForms BeepLogonDialog

A Form that hosts a BeepLogin control (themable, 9 view types — Simple, Compact, Modern, Social, Avatar, …) for the credentials surface, plus a connection ComboBox driven by BeepDataConnection.GetConnectionsSnapshot and a Test / OK / Cancel button row.

BeepLogonDialog implements IBeepLogonDialog which exposes: 

public interface IBeepLogonDialog
{
    event EventHandler<BeepLogonContext> LoggedIn;
    BeepLogonContext? LastContext { get; }
    Task<BeepLogonContext> PromptAsync(BeepLogonRequest request);
}

Wiring into BeepForms

BeepForms.LogonAsync(request) instantiates the dialog, awaits the result, and on success raises the TriggerType.WhenNewFormInstance form-level trigger with UserName and ConnectionName in TriggerContext.Parameters. 

var ctx = await myBeepForms.LogonAsync(new BeepLogonRequest {
    ConnectionName  = "northwind",
    Title           = "Sign in to Northwind",
    Prompt          = "Use your domain account"
});

Per-click hooks: OnLogin event

The dialog forwards the underlying BeepLogin.LoginClick signal as a cancellable OnLogin event. Subscribers receive a BeepLogonEventArgs that wraps a fully-populated BeepLogonContext (assembled from the live form values — no Password round-trip, no commit). Set args.Cancel = true to keep the dialog open — the status label is updated to "Login cancelled by handler."

The embedded BeepLogin is also exposed via BeepLogonDialog.LoginControl for advanced subscribers that want to wire social-login buttons, theming callbacks, or other BeepLogin-level events. 

var dialog = new BeepLogonDialog(myDataConnection);
dialog.OnLogin += (sender, args) => {
    var c = args.Context.Request;
    Trace.WriteLine($"Login attempt: {c.UserName} on {c.ConnectionName}");
    if (LooksLikeBot(c.UserName)) args.Cancel = true;
};

Usage examples

1. From a button click — built-ins

private async void btnQuery_Click(object? sender, EventArgs e)
{
    if (myBeepForms.Builtins is { } b)
    {
        b.EnterQuery();
    }
}

private async void btnExecuteQuery_Click(object? sender, EventArgs e)
{
    if (myBeepForms.Builtins is { } b)
    {
        bool ok = await b.ExecuteQueryAsync();
        if (!ok) Trace.WriteLine("Query failed");
    }
}

2. Logon at form open

protected override async void OnShown(EventArgs e)
{
    base.OnShown(e);
    var ctx = await myBeepForms.LogonAsync(new BeepLogonRequest {
        Title = "Sign in to " + myBeepForms.FormName
    });
    if (!ctx.IsSuccess) { Close(); return; }

    await myBeepForms.InitializeAsync();
}

3. Property mutation

myBeepForms.Builtins.SetItemProperty("ORDERS", "STATUS", "ENABLED", "PROPERTY_FALSE");
myBeepForms.Builtins.SetBlockMode("ORDERS", DataBlockMode.Insert);

4. LOV picker

ShowLov(block, field) loads the records for the field's LOVDefinition, then opens a modal LovPickerDialog for selection. The picker:

  • Honours LOVDefinition.Title, Width, Height, AllowSearch, ShowRowNumbers, AllowMultiSelect, AutoSizeColumns.
  • Resolves visible columns from LOVDefinition.Columns (falls back to property/field names of the first record).
  • Filters records locally by Searchable columns (LIKE %needle% via DataView.RowFilter).
  • Double-click on a row, or OK with a row selected, populates LOVSelectionResult.SelectedValue from ReturnField and DisplayValue from DisplayField (or the entire record if either is unset).
  • Cancel returns LOVSelectionResult.Cancelled() and ShowLov yields false.

Returned record is round-tripped into the block via the existing LOVSelectionResult.RelatedFieldValues dictionary (visible columns → values).
